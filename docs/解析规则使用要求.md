# 工位解析规则使用要求（P0）

## 1. 适用范围
- 页面：`视觉评估引擎 -> 导入 -> 解析验证（单文件深度查看）`
- 功能：上传单文件后，解析并展示机型、工位、工位详情、命中证据与文本片段。
- 目标：用于人工核验“工位名称解析是否正确”，并可据此持续迭代规则。

## 2. 输入要求
- 文件格式：`txt`、`md`、`csv`、`json`、`pptx`、`docx`、`xlsx`、`pdf`、`doc`、`xls`。
- 文本建议：文档中优先包含明确工位表达，例如：
  - `1工位`、`工位1`、`第3工位`
  - `STN-01`、`Station 2`、`S03`
  - `一站`、`二工位`
- 命名建议：文件名尽量携带工位信息，如 `卷绕机_1工位_外观检测.pptx`。

## 3. 解析规则（当前P0）

### 3.1 高置信度规则（编号规则）
- 数字后缀工位：`1工位`、`2站`
- 工位前缀数字：`工位1`、`工站-02`
- 英文站号：`station 3`、`stn-04`、`sta_05`
- 编号变体：`3#站`
- 中文数字：`第一工位`、`二站`、`十工位`

### 3.2 中置信度规则
- `S` 编号：`S01`、`S-2`
- 文件名命中：文件名中的 `1工位 / 2站 / station3`

### 3.3 低置信度规则（兜底）
- 当未识别到编号工位时，按关键词推断：
  - `上料` -> `上料站`
  - `检测` -> `检测站`
  - `复检` -> `复检站`
  - `扫码/条码/OCR` -> `扫码站`
  - `下料` -> `下料站`
- 若仍无结果，返回：`待确认工位`。

## 4. 编号归一化规则
- 目标格式统一为：`N站`（例如 `1站`、`2站`）。
- 归一化样例：
  - `01站`、`1#`、`工位1` -> `1站`
  - `一站`、`第一工位` -> `1站`
  - `十站` -> `10站`

## 5. 命中证据说明
预览页面会展示“命中证据”表，字段解释如下：
- `工位`：归一化后的工位编号。
- `置信度`：`高 / 中 / 低`。
- `规则`：触发该工位的解析规则名称。
- `来源`：来自正文或文件名。
- `命中片段`：原文附近片段，便于人工核查。

## 6. 使用流程（建议）
1. 上传单个协议/方案文档，点击“解析并展示细节”。
2. 先看“识别工位”和“工位详情”是否正确。
3. 再看“命中证据”是否可信（规则 + 片段）。
4. 若不正确，记录“原文表达 + 期望工位 + 当前结果”，用于新增规则。

## 7. 规则维护要求（每次优化）
每次解析规则优化必须同时完成以下动作：
1. 更新 `public/app.js` 中对应规则代码与注释。
2. 更新本文件（`docs/解析规则使用要求.md`）中的规则说明。
3. 更新 `docs/CHANGELOG.md` 版本日志。
4. 新增一份详细更新记录到 `docs/更新记录/` 目录（包含变更点、原因、影响范围、验证方式）。

## 8. 验收标准
- 同一文档重复解析，工位结果稳定一致。
- 工位编号输出统一为 `N站`。
- 预览页面可查看至少一条“命中证据”。
- 规则优化后，文档与更新记录同步齐全。

## 9. 后续开发规划（先记录后执行）

### 9.1 P1（准确率与可解释性提升）
- 目标：在保留当前 P0 能力的基础上，提高复杂文档识别准确率。
- 计划项：
  1. 增加置信度数值化评分（例如 `0~1`）并在页面展示排序。
  2. 增加“规则命中统计面板”（各规则命中次数、误判样本回溯）。
  3. 增加上下文工位名称提取优化（前后文窗口、停用词过滤）。
  4. 增加规则可配置化（规则项从代码抽离为配置对象）。
- 验收：
  - 给定样本文档集，工位识别准确率较 P0 提升。
  - 页面可按置信度排序查看工位。
  - 规则命中统计可追溯到证据片段。

### 9.2 P2（性能与工程化）
- 目标：降低重复解析耗时，减少维护成本。
- 计划项：
  1. 增加解析结果缓存（按 `name + size + lastModified`）。
  2. 增加自动保存防抖，减少频繁请求。
  3. 增加大文件保护策略（文本截断阈值、分段展示）。
  4. 增加规则单元样例集（典型命中/误判案例）。
- 验收：
  - 重复解析同文件可明显提速。
  - 高频操作下自动保存请求次数受控。
  - 大文件预览稳定，无明显卡顿。

### 9.3 执行顺序
1. 先做 P1 的评分与统计，再做规则配置化。
2. 完成 P1 后再做 P2 的缓存与性能治理。
3. 每次迭代必须同步更新：`代码注释 + 本文档 + CHANGELOG + 更新记录`。

## 10. 当前阶段验证计划（立即执行）
1. 服务启动或确认运行状态。
2. 执行脚本语法检查（`public/app.js`）。
3. 在“解析验证”界面用 2~3 份样本文档做人工核验：
   - 是否输出 `N站` 编号。
   - 是否展示命中证据（规则/来源/片段）。
   - 同一文件重复解析结果是否一致。
