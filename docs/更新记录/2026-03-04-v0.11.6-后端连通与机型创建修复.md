# 更新记录：v0.11.6 后端连通与机型创建修复

- 日期：2026-03-04
- 目标：解决“后端加载不了”和“前端创建机型失败”两项问题。

## 一、问题复现

### 1) 后端加载不了
- 用户访问 `http://localhost:3006` 出现 `ERR_CONNECTION_REFUSED`。
- 复查发现 `3006` 端口存在进程冲突（`EADDRINUSE` 场景），导致启动行为不稳定。

### 2) 前端创建机型失败
- 页面弹窗“创建机型失败，请重试”。
- 该提示过于笼统，无法区分是后端未启动、代理问题还是接口参数错误。

## 二、修复内容

### 1) 后端可观测性增强
文件：`backend/server.ts`
- 新增 `GET /api/health`：返回后端在线状态与时间戳。
- 新增 `GET /`：返回后端运行提示。

### 2) 减少误导性启动日志
文件：`backend/services/agreementService.ts`
- `initDatabase()` 中改为先执行 `PRAGMA table_info(agreements)`，仅在字段缺失时 `ALTER TABLE`。
- 避免每次启动都出现 `duplicate column name: full_text` 干扰日志。

### 3) 前端错误提示可诊断化
文件：`src/views/LibraryView.vue`
- `createMachineType()` 的异常处理增加连通性判断。
- 网络异常时提示：检查 `http://localhost:3006/api/health`。
- 保留后端 message 回传，便于快速定位真实原因。

## 三、验证结果

### 环境验证
- 前端：`http://localhost:3004/` 启动正常。
- 后端：`http://localhost:3006/api/health` 返回：
  - `ok: true`
  - `service: backend`

### 业务验证（机型创建）
- 调用：`POST /api/machine-libraries/create`
- 入参：`{ "name": "MACHINE_TEST_0304" }`
- 结果：`Machine library created successfully`，新增记录 `id: 6`。

## 四、影响范围
- 资料库页“新建机型”失败提示更清晰。
- 后端运行状态可直观检查，减少“后端是否在线”的沟通成本。
- 协议服务启动日志更干净，降低误判风险。

## 五、后续建议
1. 增加前端启动时的后端健康检查徽标（绿色/红色状态）。
2. 为端口冲突提供一键清理脚本（如 `npm run dev:reset`）。
3. 增加机型名称重复校验与友好提示。
