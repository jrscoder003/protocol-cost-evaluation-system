# 更新记录：v0.11.10 上传500复发根因修复

- 日期：2026-03-05
- 目标：彻底解决“上传失败（status code 500）”在开发运行态的复发问题。

## 一、问题分析

### 现象
- 前端上传提示：`上传失败: Request failed with status code 500`。
- 运维状态页偶发显示后端未运行，但日志中服务实际可用，定位困难。

### 根因
1. 开发阶段常用 `npm run dev` 只启动前端，后端未启动时，`Vite` 代理请求 `/api/*` 会返回 `500`。
2. `status.ps1` 依赖临时父进程 PID 判活，存在 PID 漂移，导致“在线误判离线”。

## 二、修复措施

### 1) 启动链路修复
文件：`package.json`
- `dev` 调整为同时启动前后端：
  - `concurrently "npm run dev:frontend" "npm run dev:server"`
- 新增 `dev:frontend` 保留前端单独启动能力。
- `dev:all` 同步对齐为前后端并发启动。

### 2) 运维脚本加固
文件：`scripts/start.ps1`
- 启动完成后不再记录临时父进程 PID。
- 改为按端口查询真实监听进程 PID 并写入 PID 文件。

文件：`scripts/status.ps1`
- 状态判定优先依据端口监听。
- 发现 PID 文件与端口实际进程不一致时自动纠正 PID 文件。

### 3) 前端报错可读性增强
文件：`src/services/api.ts`
- 新增 Axios 响应拦截器。
- 对“网络不可达/代理拒绝连接”场景归一化提示：
  - `后端服务不可用（请确认 3006 端口服务已启动）`
- 保留后端业务错误 `message` 透传，避免误掩盖真实后端异常。

## 三、验证结果

- 执行：`npm run dev:stop; npm run dev:start; npm run dev:status`
  - 前端：`running, URL=http://localhost:3004`
  - 后端：`running, URL=http://localhost:3006`
- 执行：`node ./scripts/test-upload-folder.mjs`
  - 返回：`Files uploaded successfully`
  - 最新记录写入成功（记录 ID 正常递增）

## 四、结论

- 本次已从“启动流程、状态检测、前端提示”三条链路完成防复发修复。
- 后续如再遇上传失败，可先用 `npm run dev:status` 快速定位是否为服务离线，而非直接归因业务逻辑异常。
